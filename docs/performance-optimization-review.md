# パフォーマンス最適化検討資料

最終更新: 2026-06-20

## 1. 目的

`https://messenger.abeto.co/` のような洗練されたブラウザゲームで使われている軽量化・高品質化の考え方を、「犬タローの虫さんまってまって」に応用できるか検討する。

本資料では、Messengerの技術要素をそのまま移植するのではなく、現在のゲーム構成に対して費用対効果が高い順に応用可能性を整理する。

2026-06-17時点では、Phase 1相当の軽量化として、スマートフォン時のDPR上限、非プレイ中の描画間引き、タブ非表示時の更新停止、背景キャッシュ、DOM更新抑制、音声処理の見直しを採用済みである。

## 2. 現在のゲーム構成

「犬タローの虫さんまってまって」は、HTML / CSS / JavaScriptで構成されたCanvas 2Dゲームである。

主な特徴:

- 描画は `canvas` の2Dコンテキストで実行
- プレイヤー、敵、虫さん、カラスの落とし物はPNG画像
- 背景画像はCanvasへキャッシュして再利用
- BGMは `HTMLAudioElement`
- 効果音はWeb Audio APIを優先し、失敗時はAudio要素へフォールバック
- リザルト保存はCanvasで共有用カード画像を生成して実行
- スマートフォン向けにスティック操作とジャンプボタンを実装

現在のアセット規模は小さく、画像はおおむね軽量である。BGMはWAV素材から `assets/bgm.mp3` へ変換済みで、画像ロード完了後に低優先で読み込み・再生開始を試みる。

## 3. Messengerから学べる主な要素

調査時点で確認できたMessenger側の特徴:

- Three.js / WebGL2ベースの3Dレンダリング
- 3D本体を分割ロード
- 端末ごとにDPRを制御
- `adaptiveDPR` による描画負荷調整
- KTX2 / Basis系の圧縮テクスチャ利用
- geometry / collision / bitmap / glyph などのWorker利用
- CDNキャッシュ、HTTP/2、modulepreloadによる配信最適化
- 低ポリゴン・スタイライズ表現により、少ない描画負荷で密度感を出している

重要なのは「高度な3Dを力任せに動かす」ことではなく、描画量、ロード量、処理タイミングを強く制御し、アートディレクションで豊かに見せている点である。

## 4. そのまま移植すべきでないもの

### WebGL / Three.js化

現時点では優先度は低い。

理由:

- 現在のゲーム性は2D Canvasで十分成立している
- 3D化すると制作コスト、当たり判定、スマホ調整、アセット制作負荷が増える
- 現在のボトルネックは3D描画性能ではない
- 手描き調の世界観はCanvas 2Dでも維持しやすい

将来的に、奥行きのあるステージ、立体的なキャラクター、回転する世界、3D演出を主軸にする場合のみ検討する。

### KTX2 / Basis圧縮テクスチャ

現時点では優先度は低い。

理由:

- KTX2 / Basisは主にWebGLテクスチャ向け
- 現在はCanvas 2DでPNGを描画している
- 画像ファイル容量も大きすぎない

代替として、PNG最適化、WebP / AVIFの追加、画像サイズの見直しを優先する。

### Worker化

現時点では必須ではない。

理由:

- 現在のオブジェクト数、衝突判定、物理計算は軽量
- Worker化するとコード構造が複雑になる
- メインスレッド負荷より、描画ピクセル数や常時アニメーションのほうが効きやすい

将来的に、敵やアイテム数が大幅に増える、マップ生成を行う、物理シミュレーションが重くなる場合に検討する。

## 5. 応用すべき要素

### 5.1 DPR制御の強化

採用済み。

スマートフォンではDPR 2から3以上の端末も多く、Canvasの実ピクセル数が大きくなりやすい。Messengerでは高DPR端末でも1.15から1.5程度に抑え、見た目と負荷のバランスを取っていた。

現在の実装:

- PC: 最大2を維持
- スマートフォンなど粗いポインター環境: 最大1.5へ制限
- 低性能端末や発熱対策を重視する場合: 最大1.25も検討

期待効果:

- 描画ピクセル数を削減
- スマートフォンでの発熱、バッテリー消費、フレーム落ちを軽減
- 画質低下は比較的小さい

優先度: 高

### 5.2 描画ループの間引き

採用済み。

現在の実装:

- ゲームプレイ中は60fps相当を維持
- タイトル画面と終了後の待機画面は30fps相当
- リザルトモーダル表示中は15fps相当
- `document.visibilityState` が `hidden` の間は更新・描画を停止し、復帰時に経過時間を補正
- 60Hz端末で誤ってフレームを間引かないよう、プレイ中の判定に許容幅を設定

期待効果:

- スマートフォンの発熱低減
- バッテリー消費の低減
- バックグラウンド時の無駄な処理削減

優先度: 高

### 5.3 背景キャッシュ方針の維持・拡張

採用済み。現在の `backgroundCache` は有効な最適化である。

推奨:

- 背景画像のキャッシュは維持
- 将来的に固定装飾、地面パターン、遠景レイヤーを増やす場合もオフスクリーンCanvasへ事前描画する
- 毎フレームのグラデーション、ベジェ、パターン描画は避ける

期待効果:

- 毎フレームのCanvas命令を削減
- 表現を増やしても描画負荷を抑えられる

優先度: 高

### 5.4 画像デコードの改善

現在は `new Image()` で画像を読み込んでいる。

推奨:

- 対応ブラウザでは `createImageBitmap` を使って画像を事前デコードする
- 非対応環境では現在の `Image()` 読み込みにフォールバックする
- まずは背景、プレイヤー、虫、敵など主要画像から対象にする

期待効果:

- 初回表示やゲーム開始直後の引っかかりを軽減
- デコード処理をブラウザにより効率よく任せられる

優先度: 中

### 5.5 アセット形式の見直し

現在の画像容量は大きすぎないが、すべて512px PNGで構成されている。

推奨:

- PNG最適化を行う
- WebP版を作り、`picture` ではなくJS側で対応確認して読み分ける
- 透明が必要なキャラクター画像はWebP losslessまたは高品質WebPを検討
- 背景画像はWebP / AVIF候補
- BGMは短いループ版や低容量版を検討

期待効果:

- 初回ロード短縮
- GitHub Pagesなど静的配信でも体感が改善

優先度: 中

### 5.6 ロード順の整理

一部採用済み。現在のゲームは画像ロード完了後にゲームループを開始し、BGMは低優先で開始している。この方針はよい。

推奨:

- ゲーム開始に必須の画像と、リザルト・共有用画像を分ける
- BGMはユーザー操作後または初回ロード安定後に取得
- 共有画像生成にだけ必要な処理は、ゲーム開始前に急いで準備しない

期待効果:

- 初回表示を軽くできる
- モバイル回線での待ち時間を減らせる

優先度: 中

### 5.7 アートで密度感を足す

Messengerの強みは、少ない描画負荷で世界が豊かに見えるアート設計にある。

「犬タロー」に応用しやすい方向:

- 遠景と近景の多層背景
- ゆっくり動く雲や草
- 前景に軽い葉っぱ・草むらシルエット
- プレイヤー着地時の地面リアクション
- 鳥の影、虫の小さな軌跡
- 画面端の柔らかい前景フレーム

注意点:

- ぼかし、強い発光、重い影は避ける
- 加算合成や大きな半透明エフェクトを増やしすぎない
- できるだけ事前描画・画像素材・小さな図形で表現する

優先度: 中

## 7. 推奨ロードマップ

### Phase 1: 軽量化の即効施策

状態: 採用済み

- スマートフォン時のDPR上限を下げる
- タブ非表示時の更新停止
- リザルト表示中の描画間引き
- 背景キャッシュ、DOM更新抑制、SP UIの軽量化
- 効果音のWeb Audio化、アイテム取得音の再生時間制限
- BGMのMP3軽量化と低優先ロード
- 変更後にPC / スマートフォン表示を確認

### Phase 2: 初回ロード体験の改善

状態: 次候補

- 画像デコード方法を改善
- WebP版アセットの検証
- BGMのロードタイミングを再確認
- GitHub Pages公開状態で初回ロードを確認

### Phase 3: 軽量な見た目強化

状態: 次候補

- 背景の多層化
- 前景草や小さな環境アニメーションの追加
- 着地、取得、鳥の動きに小さなリアクションを追加
- 追加後も発光・ぼかしに頼らない

## 8. 判断まとめ

現在のゲームに最も合う方針は、WebGL化ではなくCanvas 2Dのまま軽量化と演出密度を高めること。

Messengerから応用すべき本質は、以下の3点である。

1. 端末に合わせて描画量を抑える
2. 重い処理やアセット読み込みを必要なタイミングへ分散する
3. 技術負荷を上げるより、アート設計で豊かに見せる

この方針であれば、現在の手描き調・短時間プレイ・スマホ対応というゲームの良さを保ったまま、快適さと見た目の完成度を上げられる。
